# Telegram Lead Funnel

Локальный MVP для поиска свежих Kwork-заказов на web-разработку.

Инструмент читает веб-ленту Kwork по разработке, отбрасывает проекты с большим числом предложений, оценивает оставшиеся заказы через DeepSeek по пригодности для выполнения за неделю с AI-агентом, отправляет лиды на email и после ответа `OK <lead_id>` отправляет отклик через залогиненный Kwork Chrome.

Telegram API больше не нужен для текущей Kwork-only схемы. Telegram-источники остались как запасной режим, но основной поток идет через веб-страницы Kwork.

## Что умеет MVP

- `lead-funnel.cmd scan` — удобный запуск из Windows без переустановки пакета.
- `scan-now.cmd` — двойной клик для одного сканирования.
- `watch.cmd` — двойной клик для непрерывного мониторинга.
- `python -m app.main scan` — один раз прочитать каналы, создать лиды и отправить email.
- `python -m app.main approvals` — проверить email-ответы `OK <lead_id>` и отправить подтвержденные отклики на Kwork.
- `python -m app.main orders ...` — вести заказ после отклика: принять, взять в работу, отдать ТЗ в Codex, отправить результат на апрув вместе с готовым письмом заказчику.
- `python -m app.main order-reviews` — прочитать email-команды по заказам: `DONE <order_id>` или `FIX <order_id>: правки`.
- `python -m app.main watch` — циклически выполнять scan, approvals и order-reviews локально.

## Установка

```powershell
python -m pip install --upgrade pip
python -m pip install .[dev]
```

Если запускаешь команды из рабочей папки без переустановки пакета после правок, добавь текущий `src` в `PYTHONPATH`:

```powershell
$env:PYTHONPATH="src"
```

## Запуск из терминала

Самый удобный вариант на Windows:

```powershell
.\lead-funnel-gui.vbs
.\lead-funnel-gui.cmd
.\lead-funnel.cmd scan
.\lead-funnel.cmd watch
.\lead-funnel.cmd approvals
.\lead-funnel.cmd orders list
```

`lead-funnel-gui.vbs` открывает простое окно без консоли. `lead-funnel-gui.cmd` оставлен как запасной терминальный запуск.

- `Открыть Kwork Chrome` — открыть отдельный профиль Kwork-бота с DevTools-портом, не закрывая обычный Chrome.
- `Сканировать сейчас` — один проход по свежим заказам.
- `Старт мониторинга` — постоянная проверка по интервалу из `.env`.
- `Проверить почту` — прочитать ответы `OK <lead_id>` из почты и обработать их по старому маршруту.
- `Пересобрать отклик` — подготовить новую, более точную версию текста из карточки заказа и разобранных вложений; это только preview в редакторе.
- `OK и отправить отклик` — для выбранного лида сохранить отредактированные текст, цену, срок и название, показать финальное подтверждение и отправить отклик на Kwork.
- `Стоп мониторинга` — остановить фоновый процесс.
- нижний блок показывает живой лог.

Если `scan/watch` пишет, что Chrome DevTools port недоступен, запусти:

```powershell
.\start-kwork-browser.cmd
```

Он откроет отдельный профиль `%LOCALAPPDATA%\KworkLeadChromeUserData` с портом `9222` и откроет Kwork. Обычный Chrome не закрывается и не копируется. Если Kwork попросит вход, войди один раз именно в этом Kwork Chrome-окне: профиль сохраняется и при следующих запусках не перезатирается.

Сканер использует две рабочие вкладки в одном Chrome-профиле: вкладку ленты заказов и вкладку текущего заказа. Лента остается лентой, а данные заказа собираются со страницы конкретного заказа. Новые окна не создаются.

Для запуска двойным кликом:

- `scan-now.cmd` — один раз проверить источники и отправить новые лиды на почту.
- `watch.cmd` — оставить мониторинг включенным в открытом окне терминала.

## Настройка

1. Скопируй `.env.example` в `.env`.
2. Для текущей Kwork-only схемы оставь:
   - `TELEGRAM_CHANNELS=@freelance_dev_work`
   - можно указать `https://t.me/freelance_dev_work`, результат тот же.
3. Для режима без Telegram API оставь:
   - `TELEGRAM_API_ID=0`
   - `TELEGRAM_API_HASH=fill_later`
4. Заполни SMTP/IMAP доступы к почте.
5. Настрой лимит откликов:
   - `KWORK_MAX_RESPONSES=5` — отправлять только заказы, где на странице Kwork видно не больше 5 откликов. Счётчик проверяется и при сканировании, и прямо перед реальной отправкой.
   - `KWORK_MAX_AGE_HOURS=24` — не брать старые активные карточки; `0` отключает лимит возраста.
6. Если Kwork показывает число предложений только после входа, добавь cookie авторизованного браузера:
   - `KWORK_COOKIE=имя=значение; другое_имя=другое_значение`
   - логин и пароль в проект не нужны.
7. Основной источник сейчас Kwork web:
   - `KWORK_SOURCE=web`
   - `KWORK_PROJECTS_URL=https://kwork.ru/projects?c=11`
   - `KWORK_USE_BROWSER=1` — открыть страницу конкретного заказа через Chrome DevTools и дождаться реального счётчика `Предложений: N`.
   - `KWORK_CDP_URL=http://127.0.0.1:9222`
   - `KWORK_AUTO_REPLY=1` — после письма с `OK <lead_id>` отправлять отклик через Kwork Chrome.
   - `KWORK_LOGIN_EMAIL` и `KWORK_LOGIN_PASSWORD` — необязательный fallback: если сессия Kwork в bot Chrome слетела, скрипт попробует войти сам.
8. Не коммить `.env`: файл уже исключен через `.gitignore`.
9. Прикрепленные скриншоты, сканированные PDF и DOCX можно дополнительно разобрать через OpenRouter:
   - `OPENROUTER_API_KEY` — ключ OpenRouter.
   - `OPENROUTER_BASE_URL=https://openrouter.ai/api/v1`.
   - `OPENROUTER_VISION_MODEL=qwen/qwen3.7-plus` — модель со входом-изображением и текстовым ответом.
   - `OPENROUTER_VISION_MODE=smart` — добавляет визуальный разбор к скриншотам и к короткому или шумному OCR в сканах. `fallback` вызывает vision только когда локальный текст не найден, `off` полностью выключает облачное зрение.
   - Если переменные пустые, сканер продолжает работать без облачного зрения.

## Безопасный поток

1. Инструмент читает свежую веб-ленту Kwork: обычные карточки, а если интерфейс Kwork ещё не отрисовал их, встроенные данные самой страницы.
2. Создает лид в SQLite.
3. Инструмент открывает страницу заказа и добавляет к анализу полное описание, видимый текст страницы и найденные вложения/файлы.
4. DeepSeek оценивает заказ: сложность, срок, цену, пригодность, главную боль клиента, короткий план работ и риски.
5. Для подходящего заказа отдельный DeepSeek-проход составляет и проверяет отклик: он опирается на задачу и ТЗ, называет конкретные действия и не пропускает цену или общие фразы. Обычные детали в первом сообщении не запрашиваются; один вопрос допустим только когда AI-оценка признала его критичным для старта. Стоимость хранится отдельно для поля Kwork и не вставляется в сообщение заказчику.
6. Для старого или слабого черновика выбери лид и нажми `Пересобрать отклик`. Новый текст появится только в редакторе, где его можно проверить и поправить; он не сохраняется и не отправляется сам.
7. Для отправки из приложения выбери лид, при необходимости поправь текст, цену, срок и название, затем нажми `OK и отправить отклик`. Перед окном подтверждения приложение повторно сверяет текст с карточкой и ТЗ: старый или выдумывающий несуществующую работу черновик не будет сохранён или отправлен, а окно предложит пересобрать либо поправить его. После успешной проверки и `Да` отклик реально уходит на Kwork.
8. Почтовый маршрут остаётся доступен: ответь на письмо одной строкой `OK <lead_id>`, затем `Проверить почту` или включенный мониторинг сначала повторно сверит текст с карточкой и ТЗ. Если черновик старый, общий или обещает несуществующую работу, отклик не уйдёт: причина появится у лида, его можно пересобрать или поправить и подтвердить новым `OK`. Безопасный текст откроет заказ в Kwork Chrome, заполнит текст отклика, цену, срок и название и нажмёт отправку.

Без явного подтверждения в приложении или `OK <lead_id>` из почты система ничего не отправляет. После успешной отправки лид блокируется от повторного отклика; пока отправка выполняется, кнопку нельзя нажать второй раз.
Кнопка `Проверить заказ` обновляет из живой страницы Kwork число предложений и время проверки, не заполняя и не отправляя форму. `Проверить свежие` делает то же для активной очереди, максимум для 30 неотправленных лидов за проход, начиная с самых свежих и наименее конкурентных. Если последний результат уже больше `KWORK_MAX_RESPONSES`, `OK и отправить отклик` сразу остановит попытку до открытия формы; повторная проверка происходит и внутри отправителя.
Перед заполнением Kwork текст отклика проверяется еще раз: даже у старых лидов цена, бюджет и условия оплаты не попадут в сообщение заказчику, а сохранятся только в полях формы. Явная техническая задача про прием оплаты на сайте остается в контексте и может быть упомянута в отклике без добавления коммерческих условий.
Непосредственно перед заполнением формы система снова открывает страницу заказа через Kwork Chrome. Если заказ закрыт, поле `Предложений` не читается или откликов стало больше `KWORK_MAX_RESPONSES`, сообщение не отправляется, а причина остаётся у лида.

Сначала работает быстрый фильтр: сканер берёт свежие карточки из страницы Kwork, отбрасывает записи старше `KWORK_MAX_AGE_HOURS`, а затем для каждого проекта открывает его страницу и ждёт видимый текст `Предложений: N`. Если предложений больше `KWORK_MAX_RESPONSES` или число не удалось прочитать, письмо не отправляется. Число услуг исполнителя (`kwork_count`) не используется вместо числа предложений. Затем rule-фильтр применяет Bitrix/Битрикс, заданные тобой стоп-слова и необязательный список обязательных слов. После этого DeepSeek оценивает остальные заказы по полному тексту, вложениям, сложности, score и сроку не больше `LEAD_MAX_DAYS`. Поле `LEAD_HARD_REJECT_KEYWORDS` оставлено как необязательный ручной предфильтр: по умолчанию оно пустое, чтобы не терять потенциально выполнимые заказы до AI-оценки.

В GUI есть блок `Настройки отбора`. Он сохраняет значения в `.env`, не трогая пароли и ключи:

- `KWORK_MAX_RESPONSES` — максимум откликов на Kwork-заказе.
- `KWORK_MAX_AGE_HOURS` — максимальный возраст карточки Kwork; `0` оставляет возраст без фильтра.
- `SCAN_INTERVAL_SECONDS` — как часто мониторинг обновляет ленту.
- `MAX_POSTS_PER_CHANNEL` — сколько свежих карточек брать за проход.
- `LEAD_MIN_SCORE` — минимальная оценка AI от 0 до 100.
- `LEAD_MAX_DAYS` — максимальный срок выполнения, который считаем подходящим.
- `LEAD_ACCEPT_DECISIONS` — какие решения AI принимать, обычно `accept, maybe`.
- `LEAD_BLOCKED_KEYWORDS` — жесткие стоп-слова, например `битрикс, bitrix`.
- `LEAD_HARD_REJECT_KEYWORDS` — дополнительные жёсткие стоп-слова; по умолчанию пусто.
- `LEAD_REQUIRED_KEYWORDS` — если заполнить, заказ должен содержать хотя бы одно слово из списка.
- `KWORK_PROJECTS_URL` — страница Kwork, которую мониторим.

Если мониторинг уже запущен, после сохранения настроек останови и запусти его заново, чтобы новый процесс перечитал `.env`.

В GUI есть блок `Лиды и отклик`. Он показывает последние лиды из SQLite, их статус, score, цену, срок и краткое название задачи. По выбранному лиду можно:

- работать с активной очередью за `KWORK_MAX_AGE_HOURS`; `Показать архив` возвращает старые лиды без удаления истории;
- ориентироваться по колонке `Срочность`: `Срочно` означает свежий заказ с 0-2 живыми откликами, `Высокий` - свежий заказ в пределах лимита, `Проверить` - Kwork пока не отдал счетчик, `Стоп` - заказ уже отправлен, отклонен AI или перерос лимит. Очередь автоматически ставит самые быстрые возможности выше;
- открыть заказ в отдельном Kwork Chrome;
- обновить живое число откликов через `Проверить заказ`;
- нажать `Переоценить AI`, чтобы заново прочитать страницу Kwork, новые вложения и ТЗ, а затем получить score, боль клиента, план работ и риски. Кнопка работает только в режиме просмотра: не открывает форму отклика, не отправляет сообщение и не перезаписывает вручную исправленные название, цену, срок или текст. `KWORK-ДАННЫЕ` и отчёт ТЗ сохраняются в новой сводке; если Kwork временно не показал старый файл, последний успешный отчёт остаётся в карточке. Решение `reject` останавливает отправку до формы Kwork;
- поправить название, цену, срок и текст отклика;
- сразу увидеть в строке статуса, что сохраненный черновик требует правки и почему;
- пересобрать текст по текущим данным задачи и вложений, не меняя базу до `Сохранить`;
- сохранить все правки в базе, чтобы они сохранились после перезапуска и использовались при `OK <lead_id>`;
- заполнить форму Kwork без отправки;
- отправить отклик в Kwork и перевести лид в `sent`.

Ряд этих действий расположен сразу под названием, ценой и сроком лида, поэтому `Пересобрать отклик` и `OK и отправить отклик` остаются видны даже при длинной карточке или большом числе вложений.

Если отправка упала, лид получает статус `failed`, а причина сохраняется в поле ошибки и показывается при выборе лида. Такой лид можно открыть, поправить и отправить повторно из GUI.

Под деталями лида есть блок `Вложения и отработка`. В нем видно имя файла, статус обработки, тип, локальную копию и краткое содержание. Кнопки рядом позволяют открыть скачанный файл, открыть исходную ссылку Kwork или скопировать полный отчет по выбранному вложению. Новые вложения сохраняются в SQLite и в папку `data/attachments/post_<post_id>/`; старые лиды без отдельных записей показываются через fallback из блока `ФАЙЛЫ/ТЗ` в summary.

Текст в блоках `Данные заказа и AI-оценка`, `Текст отклика` и `Лог` можно выделять мышью и копировать через `Ctrl+C` или `Ctrl+Insert`.

В письме и в левой панели лида есть блок `AI-ОЦЕНКА`: решение, score, сложность, срок, цена, главная боль клиента, короткий план работ, причины, риски и максимум одно деловое уточнение только при реальной блокировке старта. Если облачный AI временно недоступен, карточка всё равно получает осторожный план по тексту заказа, без выдуманных интеграций и доступов. Если на странице Kwork найдены важные поля, ниже появляется `KWORK-ДАННЫЕ`: бюджет, остаток времени, число предложений, покупатель/процент найма. Черновик отклика пишет человеческим тоном: показывает понимание задачи, называет конкретные действия, срок и ожидаемый результат, не придумывая состояние сайта или устройства, которого нет в ТЗ, и не оценивая навыки заказчика. Цена в текст отклика не попадает.

Если к заказу прикреплены ТЗ, документы или скриншоты, инструмент сначала локально скачивает и разбирает файлы, а в DeepSeek передает уже извлеченный текст, названия, ссылки и статус обработки. DeepSeek не получает бинарные `pdf`/`docx` напрямую. Сейчас автоматически читаются небольшие `txt`, `md`, `csv`, `html`, `json`, `xml`, `docx` и `pdf` до 2 MB. `zip`-архивы читаются в памяти; для `rar` используется консольный `UnRAR`/`RAR` из WinRAR, а для `7z` - найденный `7z.exe`. Во всех архивах сначала берется список файлов, затем DeepSeek выбирает до восьми полезных файлов для чтения, а при отсутствии ключа работает fallback по названиям ТЗ/брифов/макетов. Вся распаковка идет потоком в память с лимитом на каждый файл: архив целиком в папку проекта не извлекается. Скриншоты `png`, `jpg`, `jpeg`, `webp` прогоняются через локальный Tesseract OCR из `D:\Tesseract-OCR\tesseract.exe`. В режиме `smart` OpenRouter дополняет скриншоты визуальным описанием, а PDF/DOCX с коротким или сомнительным текстом получает второй визуальный проход. В лиде появляется статус `OCR + vision прочитан`, чтобы было видно, что именно реально прочитано.

Если Kwork не дает открыть файлы без авторизации, нажми `Открыть Kwork Chrome` и войди в Kwork в открывшемся отдельном окне. Сканер сначала пробует обычное скачивание, а при отказе или HTML-заглушке вместо файла берет cookies из этой залогиненной Chrome-сессии и повторяет скачивание.

В письме в блоке `AI-ОЦЕНКА` дополнительно появляется `ФАЙЛЫ/ТЗ`: для каждого файла видно, скачался он или нет, прочитался ли документ, открылся ли архив, сработал ли OCR и краткое содержание найденного текста. Если `UnRAR`/`RAR` для RAR или `7z.exe` для 7Z недоступен, архив поврежден либо защищен паролем, лид остается рабочим, а в отчете появляется понятная причина, почему архив не открыт. Пути к утилитам при необходимости задаются локально через `UNRAR_EXE` и `SEVEN_ZIP_EXE`.

Tesseract настроен на диск `D:`. Если нужно переопределить путь, используй только `D:`:

```dotenv
TESSERACT_CMD=D:\Tesseract-OCR\tesseract.exe
```

Дедупликация идет по `channel="kwork-web"` и ID проекта Kwork. Если перезапустить `scan` или `watch`, уже отправленный project id не создаст второй лид.

Для Kwork web-режима скрипт использует Chrome через DevTools. Кнопка запускает отдельный профиль с портом `9222`, чтобы сохранить Kwork-вход бота и не трогать обычный Chrome.

Если Kwork Chrome не залогинен, `approvals` не отправит отклик и пометит лид как `failed`; в открытом Kwork Chrome нужно один раз войти в аккаунт и повторить поток на следующем лиде.
Если заказ уже закрыт, удалён или Kwork показывает «Страница не найдена», сканер не создаст по нему лид, а уже существующая карточка получит понятную причину ошибки сразу, без долгого ожидания несуществующего поля отклика.

## Поток выполнения заказов

После успешной отправки отклика лид автоматически становится заказом с коротким названием из карточки Kwork, а не со служебной AI-оценкой. В ручном режиме заказ можно добавить самому:

```powershell
python -m app.main orders receive --contact "@client_dev" --title "Лендинг" --brief "Сверстать HTML/CSS/JS лендинг"
python -m app.main orders start 1
python -m app.main orders handoff 1
python -m app.main orders submit 1 --deliverable "Готовая ссылка: https://example.com"
python -m app.main order-reviews
python -m app.main orders list
```

Статусы заказа:

- `received` — заказ получен.
- `in_progress` — заказ взят в работу.
- `ready_for_approval` — результат отправлен на проверку; письмо содержит предложение/сообщение заказчику, собранное из ТЗ, результата и последних правок.
- `revision_requested` — пришли правки, нужно доработать и снова выполнить `orders submit`.
- `done` — заказ одобрен командой `DONE <order_id>`.

Команды в ответе на письмо с результатом:

- `DONE <order_id>` — принять работу и пометить заказ сделанным.
- `FIX <order_id>: что поправить` — вернуть заказ в правки.

## Передача заказа в Codex

Команда handoff создает markdown-файл с готовым заданием для Codex:

```powershell
python -m app.main orders handoff 1
```

По умолчанию файл создается в `handoffs/order-1-handoff.md`. В него попадают:

- ID, статус, контакт и даты заказа.
- ТЗ заказчика.
- Последние правки, если они есть.
- Последний результат, если заказ уже отдавался на проверку.
- Definition of Done и минимальная команда проверки.

Папка `handoffs/` исключена из git, потому что эти файлы могут содержать пользовательские детали заказа.
